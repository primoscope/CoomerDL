### When to Update Documentation

1. **README.md** - Update when:
   - New major features are added
   - Installation process changes
   - Supported sites change significantly

2. **ROADMAP.md** - Update when:
   - Features are completed
   - New features are planned
   - Priorities change

3. **DEVELOPMENT_ROADMAP.md** - Update when:
   - Tasks are completed
   - New technical debt identified
   - Architecture decisions made

4. **Implementation Guides** - Update when:
   - Major features implemented
   - Architecture changes occur
   - Best practices evolve

### Documentation Review Schedule

- **Weekly**: Check implementation summaries

- **Quarterly**: Audit all documentation for accuracy
- **Per Release**: Update README, ROADMAP, and guides

## üìù Contributing to Documentation

### Style Guide
- Use Markdown format
- Include table of contents for long documents
- Use emojis for visual navigation
- Keep language clear and concise
- Include code examples where relevant

### File Naming
- Use SCREAMING_SNAKE_CASE for guides (e.g., `BUILDING.md`)
- Use Title_Case for summaries (e.g., `Session_Summary.md`)
- Prefix with category if needed (e.g., `PHASE4_SUMMARY.md`)

### Organization
- `/planning/` - Future-focused, what we're building
- `/implementation/` - Present-focused, what we've built
- `/development/` - Process-focused, how we build
- `/phases/` - Historical, what we built when
- `/session_summaries/` - Logs, what happened in sessions
